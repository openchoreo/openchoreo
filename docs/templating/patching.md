# Patching

This guide explains how to use the patching system in OpenChoreo Traits to modify resources generated by ComponentTypes. The patching engine extends standard JSON Patch (RFC 6902) with powerful features like array filtering and automatic parent creation.

## Overview

Traits can modify existing resources using patches, which are JSON Patch operations enhanced with:
- Array filtering using JSONPath-like syntax
- Automatic parent map creation
- CEL-based resource targeting
- Shallow merge operations
- forEach iteration support

## Basic Patch Structure

Patches are defined in the Trait's `spec.patches` section:

```yaml
apiVersion: v1alpha1
kind: Trait
metadata:
  name: monitoring-sidecar
spec:
  patches:
    - target:
        group: apps
        version: v1
        kind: Deployment
      operations:
        - op: add
          path: /spec/template/spec/containers/-
          value:
            name: prometheus-exporter
            image: prom/node-exporter:latest
            ports:
            - containerPort: 9100
```

## Supported Operations

### add
Adds a value at the specified path. Creates parent maps automatically if they don't exist.

```yaml
# Add a new label
- op: add
  path: /metadata/labels/monitoring
  value: "enabled"

# Append to array
- op: add
  path: /spec/containers/-
  value:
    name: sidecar
    image: sidecar:latest

# Create nested structure (auto-creates parents)
- op: add
  path: /metadata/annotations/example.com~1version
  value: "v2.0"
```

### replace
Replaces an existing value. The path must exist.

```yaml
# Replace a value
- op: replace
  path: /spec/replicas
  value: 3

# Replace array element
- op: replace
  path: /spec/containers/0/image
  value: nginx:latest
```

### remove
Removes a value at the path. No-op if path doesn't exist (idempotent).

```yaml
# Remove a label
- op: remove
  path: /metadata/labels/deprecated

# Remove array element
- op: remove
  path: /spec/containers/1
```

### mergeShallow
Overlays top-level keys while preserving siblings (not standard RFC 6902).

```yaml
# Merge labels without removing existing ones
- op: mergeShallow
  path: /metadata/labels
  value:
    app: my-app
    version: v2

# Merge container environment variables
- op: mergeShallow
  path: /spec/template/spec/containers/0/env
  value:
    - name: NEW_VAR
      value: new-value
```

## Array Filtering

Use JSONPath-like syntax to target specific array elements:

### Basic Filtering

```yaml
# Target container by name
- op: add
  path: /spec/template/spec/containers[?(@.name=='app')]/env/-
  value:
    name: MONITORING
    value: enabled

# Target volume by name
- op: replace
  path: /spec/template/spec/volumes[?(@.name=='data')]/emptyDir
  value:
    sizeLimit: 10Gi
```

### Multiple Conditions

```yaml
# Target containers with specific image
- op: add
  path: /spec/containers[?(@.image=='nginx:latest')]/ports/-
  value:
    containerPort: 8080

# Target by multiple fields
- op: replace
  path: /spec/containers[?(@.name=='app' && @.image=='myapp:v1')]/image
  value: myapp:v2
```

### Complex Filters

```yaml
# Target containers with specific port
- op: add
  path: /spec/containers[?(@.ports[0].containerPort==8080)]/env/-
  value:
    name: HTTP_PORT
    value: "8080"

# Target by nested field
- op: add
  path: /spec/volumes[?(@.persistentVolumeClaim.claimName=='data')]/persistentVolumeClaim/readOnly
  value: true
```

## Path Resolution Behavior

Understanding how paths are resolved is crucial:

| Path Type | Operation | Zero Matches Behavior |
|-----------|-----------|----------------------|
| **Filter** `[?(...)]` | add, replace, remove | **Error** - Explicit selector must match |
| **Non-filter** | add, replace | **No-op** - Silent skip if parent missing |
| **Non-filter** | remove | **No-op** - Idempotent removal |

```yaml
# This will ERROR if no container named 'app' exists
- op: add
  path: /spec/containers[?(@.name=='app')]/env/-
  value: {name: VAR, value: val}

# This will be no-op if /spec/optional doesn't exist
- op: add
  path: /spec/optional/field
  value: something

# This is always safe (idempotent)
- op: remove
  path: /spec/unwanted
```

## Resource Targeting

### Basic Targeting

```yaml
patches:
  - target:
      group: apps        # API group (empty for core resources)
      version: v1        # API version
      kind: Deployment   # Resource kind
      name: my-app       # Optional: specific resource name
    operations:
      - op: add
        path: /metadata/labels/patched
        value: "true"
```

### CEL-Based Filtering

Use `where` clause with CEL expressions to target resources conditionally:

```yaml
patches:
  - target:
      group: apps
      version: v1
      kind: Deployment
      where: ${resource.spec.replicas > 1}  # Only multi-replica deployments
    operations:
      - op: add
        path: /metadata/annotations/ha-mode
        value: "true"

  - target:
      kind: Service
      where: ${resource.spec.type == 'LoadBalancer'}
    operations:
      - op: add
        path: /metadata/annotations/external
        value: "true"
```

## ForEach Iteration

Apply patches iteratively over a list:

```yaml
patches:
  - target:
      kind: ConfigMap
    forEach: ${parameters.environments}
    var: env
    operations:
      - op: add
        path: /data/${env.name}
        value: ${env.value}

  - target:
      group: apps
      version: v1
      kind: Deployment
    forEach: ${parameters.extraPorts}
    var: port
    operations:
      - op: add
        path: /spec/template/spec/containers[?(@.name=='app')]/ports/-
        value:
          containerPort: ${port.number}
          name: ${port.name}
```

## RFC 6901 Escaping

Special characters in paths must be escaped:

- `~` becomes `~0`
- `/` becomes `~1`

```yaml
# Path with / character (Kubernetes annotation)
- op: add
  path: /metadata/annotations/kubernetes.io~1ingress-class
  value: nginx

# Path with ~ character
- op: add
  path: /data/config~0backup
  value: backup-data

# Complex example with both
- op: add
  path: /metadata/labels/app.kubernetes.io~1managed~0by
  value: openchoreo
```

## Advanced Examples

### Add Sidecar Container

```yaml
apiVersion: v1alpha1
kind: Trait
metadata:
  name: logging-sidecar
spec:
  patches:
    - target:
        group: apps
        version: v1
        kind: Deployment
      operations:
        # Add sidecar container
        - op: add
          path: /spec/template/spec/containers/-
          value:
            name: fluentd
            image: fluent/fluentd:v1.14
            volumeMounts:
            - name: logs
              mountPath: /var/log

        # Add shared volume
        - op: add
          path: /spec/template/spec/volumes/-
          value:
            name: logs
            emptyDir: {}

        # Mount log volume to main container
        - op: add
          path: /spec/template/spec/containers[?(@.name=='${metadata.name}')]/volumeMounts/-
          value:
            name: logs
            mountPath: /app/logs
```

### Modify Service Ports

```yaml
apiVersion: v1alpha1
kind: Trait
metadata:
  name: metrics-port
spec:
  patches:
    - target:
        kind: Service
      operations:
        # Add metrics port to service
        - op: add
          path: /spec/ports/-
          value:
            name: metrics
            port: 9090
            targetPort: metrics
            protocol: TCP

        # Add selector for metrics
        - op: mergeShallow
          path: /spec/selector
          value:
            metrics: enabled
```

### Conditional Resource Limits

```yaml
apiVersion: v1alpha1
kind: Trait
metadata:
  name: resource-optimizer
spec:
  schema:
    tier: "string | enum=small,medium,large default=small"

  patches:
    - target:
        group: apps
        version: v1
        kind: Deployment
        where: ${parameters.tier == 'small'}
      operations:
        - op: mergeShallow
          path: /spec/template/spec/containers[?(@.name=='${metadata.name}')]/resources
          value:
            limits:
              memory: 512Mi
              cpu: 500m
            requests:
              memory: 256Mi
              cpu: 250m

    - target:
        group: apps
        version: v1
        kind: Deployment
        where: ${parameters.tier == 'large'}
      operations:
        - op: mergeShallow
          path: /spec/template/spec/containers[?(@.name=='${metadata.name}')]/resources
          value:
            limits:
              memory: 4Gi
              cpu: 2000m
            requests:
              memory: 2Gi
              cpu: 1000m
```

### Network Policy Addition

```yaml
apiVersion: v1alpha1
kind: Trait
metadata:
  name: network-isolation
spec:
  patches:
    # First ensure NetworkPolicy exists via 'creates'
    # Then patch it based on configuration
    - target:
        group: networking.k8s.io
        version: v1
        kind: NetworkPolicy
        name: ${metadata.name}-netpol
      operations:
        # Add ingress rules
        - op: add
          path: /spec/ingress/-
          value:
            from:
            - podSelector:
                matchLabels:
                  app: frontend
            ports:
            - protocol: TCP
              port: 8080

        # Add egress rules
        - op: add
          path: /spec/egress/-
          value:
            to:
            - podSelector:
                matchLabels:
                  app: database
            ports:
            - protocol: TCP
              port: 5432
```

## Common Patterns

### Safe Label Addition

```yaml
# Safely add labels without overwriting existing ones
- op: mergeShallow
  path: /metadata/labels
  value:
    monitoring: enabled
    trait: ${metadata.name}
```

### Conditional Container Environment

```yaml
# Add environment variables based on configuration
- target:
    group: apps
    version: v1
    kind: Deployment
  forEach: ${parameters.envVars}
  var: envVar
  operations:
    - op: add
      path: /spec/template/spec/containers[?(@.name=='${metadata.name}')]/env/-
      value:
        name: ${envVar.name}
        value: ${envVar.value}
```

### Volume Mount Injection

```yaml
# Add volume mounts to specific containers
- op: add
  path: /spec/template/spec/volumes/-
  value:
    name: config
    configMap:
      name: ${metadata.name}-config

- op: add
  path: /spec/template/spec/containers[?(@.name=='app')]/volumeMounts/-
  value:
    name: config
    mountPath: /etc/config
    readOnly: true
```

### Service Mesh Annotations

```yaml
# Add Istio sidecar annotations
- op: mergeShallow
  path: /spec/template/metadata/annotations
  value:
    sidecar.istio.io/inject: "true"
    sidecar.istio.io/proxyCPU: "100m"
    sidecar.istio.io/proxyMemory: "128Mi"
```

## Best Practices

### 1. Use Filters for Explicit Targeting

```yaml
# Good - explicit container selection
- op: add
  path: /spec/containers[?(@.name=='app')]/env/-
  value: {name: VAR, value: val}

# Bad - assumes container position
- op: add
  path: /spec/containers/0/env/-
  value: {name: VAR, value: val}
```

### 2. Prefer mergeShallow for Maps

```yaml
# Good - preserves existing labels
- op: mergeShallow
  path: /metadata/labels
  value:
    new-label: value

# Bad - replaces all labels
- op: replace
  path: /metadata/labels
  value:
    new-label: value
```

### 3. Handle Missing Paths Gracefully

```yaml
# Good - check existence with CEL
- target:
    kind: Deployment
    where: ${has(resource.spec.template.spec.nodeSelector)}
  operations:
    - op: mergeShallow
      path: /spec/template/spec/nodeSelector
      value:
        disk: ssd

# Bad - assumes field exists
- op: replace
  path: /spec/template/spec/nodeSelector
  value:
    disk: ssd
```

### 4. Use Proper Escaping

```yaml
# Good - escaped annotation
- op: add
  path: /metadata/annotations/example.com~1version
  value: v1

# Bad - unescaped will fail
- op: add
  path: /metadata/annotations/example.com/version
  value: v1
```

### 5. Test Array Filters

```yaml
# Good - specific filter
- op: add
  path: /spec/containers[?(@.name=='app' && @.image contains 'myapp')]/env/-
  value: {name: VERSION, value: v2}

# Bad - ambiguous filter might match multiple
- op: replace
  path: /spec/containers[?(@.ports)]/image
  value: new-image
```

## Error Handling

### Common Errors and Solutions

**"no matches found for array filter"**
- Cause: Filter didn't match any array elements
- Solution: Verify filter criteria or use conditional targeting with `where`

**"path does not exist"**
- Cause: Trying to replace non-existent path
- Solution: Use `add` instead or create parent first

**"invalid array index"**
- Cause: Array index out of bounds
- Solution: Use filters or `-` for append

**"cannot create array element by index"**
- Cause: Trying to create array element at specific index
- Solution: Use `-` to append or ensure array exists first

## Debugging Tips

1. **Start with simple patches** and gradually add complexity
2. **Use targeting conditions** to apply patches only when appropriate
3. **Test with actual resources** to verify filter behavior
4. **Check escaped characters** in paths match actual field names
5. **Verify array filters** match expected elements

## Complete Example

Here's a comprehensive Trait using various patching techniques:

```yaml
apiVersion: v1alpha1
kind: Trait
metadata:
  name: production-hardening
spec:
  schema:
    securityContext:
      runAsNonRoot: "boolean | default=true"
      fsGroup: "integer | default=1000"

    monitoring:
      enabled: "boolean | default=true"
      port: "integer | default=9090"

    resources:
      tier: "string | enum=small,medium,large default=medium"

  patches:
    # Add security context
    - target:
        group: apps
        version: v1
        kind: Deployment
      operations:
        - op: mergeShallow
          path: /spec/template/spec/securityContext
          value:
            runAsNonRoot: ${parameters.securityContext.runAsNonRoot}
            fsGroup: ${parameters.securityContext.fsGroup}
            runAsUser: 1000

    # Add monitoring port to containers
    - target:
        group: apps
        version: v1
        kind: Deployment
        where: ${parameters.monitoring.enabled}
      operations:
        - op: add
          path: /spec/template/spec/containers[?(@.name=='${metadata.name}')]/ports/-
          value:
            name: metrics
            containerPort: ${parameters.monitoring.port}

    # Add prometheus annotations to pods
    - target:
        group: apps
        version: v1
        kind: Deployment
        where: ${parameters.monitoring.enabled}
      operations:
        - op: mergeShallow
          path: /spec/template/metadata/annotations
          value:
            prometheus.io/scrape: "true"
            prometheus.io/port: "${parameters.monitoring.port}"
            prometheus.io/path: "/metrics"

    # Set resource limits based on tier
    - target:
        group: apps
        version: v1
        kind: Deployment
        where: ${parameters.resources.tier == 'medium'}
      operations:
        - op: mergeShallow
          path: /spec/template/spec/containers[?(@.name=='${metadata.name}')]/resources
          value:
            limits:
              memory: 2Gi
              cpu: 1000m
            requests:
              memory: 1Gi
              cpu: 500m

    # Add network policy labels
    - target:
        kind: Service
      operations:
        - op: mergeShallow
          path: /spec/selector
          value:
            security: hardened
            monitoring: ${parameters.monitoring.enabled ? "enabled" : "disabled"}

    # Add disruption budget reference
    - target:
        group: apps
        version: v1
        kind: Deployment
      operations:
        - op: add
          path: /metadata/annotations/policy.openchoreo.io~1disruption-budget
          value: ${metadata.name}-pdb
```
